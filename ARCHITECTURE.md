# Architecture Decisions

## ADR-001: SQLite through GRDB.swift

**Decision:** Use GRDB.swift 7.x over the system SQLite library, with schema changes registered by one `DatabaseMigrator` factory.

**Why:** GRDB provides explicit migrations, safe connection ownership, and first-class SQLite/FTS5 support without introducing an object graph framework. Phase 0 uses a single `DatabaseQueue`; later work should change that only when measured concurrency needs justify it.

## ADR-002: Local-only application boundary

**Decision:** The MVP has no account, network, cloud, or synchronization dependency. The App Sandbox entitlement is enabled without network or user-selected-file capabilities.

**Why:** Launch and local text capture must not depend on a service or permission that the current phase does not need. Attachment and link-preview phases will add narrowly scoped capabilities only when required.

## ADR-003: Separate database and managed-file storage

**Decision:** Store note metadata in `notes.sqlite` under the app's Application Support root. Future originals, thumbnails, preview images, and staging files belong under sibling managed directories rather than inside the database.

**Why:** SQLite remains responsible for transactional metadata and search. Large files retain ordinary filesystem semantics and can later use atomic staging and integrity checks. Phase 0 creates only the root and database because attachment directories have no current behavior.

## ADR-004: Injectable storage root

**Decision:** Resolve the production Application Support URL once at composition time and inject an explicit root URL in tests.

**Why:** No persistence type relies on a global mutable singleton, and automated tests cannot accidentally modify the user's archive.

## ADR-005: macOS 14 deployment target

**Decision:** Keep the proposed macOS 14 minimum. GRDB 7.11.1 requires Swift 6.1 and Xcode 16.3, but does not require a newer macOS deployment target.

**Why:** macOS 14 supplies the SwiftUI APIs used by the placeholder UI and matches the accepted plan. The local machine has Swift 6.3 command-line tools but no full Xcode installation; that is a verification limitation, not a product reason to raise the deployment target.

## ADR-006: Blob ownership and logical attachments

**Decision:** `attachment_blobs` owns one immutable managed original, its optional regenerable thumbnail, SHA-256, byte size, validated media classification, and dimensions. `attachments` owns each logical note relationship, original filename, creation time, and order. Equal bytes may therefore share one blob while remaining separate attachments, including when their original filenames differ. Completed composer imports are recorded in `draft_attachments` until send commits. The current one-row draft schema is paired with one application `Window`; a send fails closed unless the supplied attachment IDs exactly match the persisted manifest.

**Why:** Deduplication must not erase user-visible attachment identity, and deleting one note must not remove bytes still referenced by another note. Persisting completed staging records also keeps an attachment draft and its text together across relaunch instead of silently degrading a failed send into a text-only draft.

## ADR-007: Database/filesystem commit boundary

**Decision:** Finalization, startup maintenance, missing-blob repair, and permanent-delete cleanup use one library-scoped coordinator. Imports use unique IDs and can begin only after startup maintenance completes; each streams into staging while hashing. Startup first reads and represents the persisted attachment manifest as a required operation, then performs filesystem cleanup as best effort. A manifest read failure leaves composition unavailable with Retry; a later cleanup failure reports an error without hiding or deleting manifested drafts. Send validates staged and existing bytes, hard-links each new blob into its final same-volume location, synchronizes the final directory, and only then runs one SQLite transaction that creates the note, blob rows, logical attachment rows, clears the text draft, and removes exactly the matching attachment manifest rows. Staging remains intact until that transaction commits. Cancel/retry keeps one active generation per attachment ID so an obsolete task cannot remove a replacement attempt. Removing an active row invalidates that generation but retains and awaits the task until its database/file cleanup settles; Send remains unavailable during that interval. All pending mutation is disabled while send owns the captured draft.

If SQLite fails, only final links created for that attempt are removed; the manifested staging copies remain retryable. A matching staged file may repair a missing referenced original before send, and that repair is retained even if the new note fails. Once SQLite commits, the note is successful even if post-commit staging removal fails. Startup removes unmanifested staging and managed files with no blob row, retains manifested staging, and reports missing referenced originals without deleting metadata.

Whole-draft discard first cancels and settles active imports, then atomically deletes the text draft and attachment manifest in SQLite. UI clears only after that commit. If the transaction fails, the still-visible text is synchronously persisted because its prior debounce was canceled; failure of that recovery save is reported explicitly so the user can copy it. Returned staging paths are removed afterward; a failed physical removal leaves an unmanifested file that startup can safely remove. Permanent deletion takes the inverse boundary: SQLite first deletes the trashed note and only blob rows with no remaining logical references, then returns those paths for physical removal while the coordinator is still held. A failed physical removal leaves an unreferenced orphan that startup can safely remove.

External open, reveal, and copy operations receive independent presentation copies rather than hard links to managed blobs. Quick Look alone reads the managed original directly. Export streams to the exact `NSSavePanel` URL while its security scope is active because the sandbox extension does not authorize a random sibling; an interrupted replacement may leave partial destination data and is reported as such. These boundaries prevent other applications from mutating deduplicated archive bytes.

**Recovery invariant:** SQLite never commits an attachment path before immutable final bytes exist. A process interruption may leave extra unreferenced files, but must not create a newly committed note whose final original was never installed. Missing bytes for older committed metadata are reported, never silently discarded or replaced with unverified content.

**Why:** SQLite and the filesystem cannot share one atomic transaction. Immutable staging plus install-before-metadata makes the unavoidable partial states deterministic, recoverable, and safe for reference-counted deduplication.

## ADR-008: Persisted, opt-in link-preview work

**Decision:** URL detection and note persistence stay on the local note-write path; network work does not. Every body-edit transaction increments `notes.linkPreviewRevision`, making all previous associations ineligible at the same durable boundary as the new text. Production edits run through `LinkPreviewCoordinator`; on that actor, the body edit, revision-aware URL reconciliation, image cleanup, cancellation, and final hydrated read form one non-suspending operation. Reconciliation snapshots both body and revision and stamps `link_previews.reconciledRevision` only if both still match. Work selection, final pre-fetch eligibility, status/cache commits, hydration, FTS metadata, and **Has Link** all reject a revision mismatch. This prevents a work item whose earlier eligibility check raced an edit from invoking the fetch after that edit has removed its URL.

A separate `link_preview_cache` row owns shared metadata, refresh timestamps, request-key retry deadline, and the optional managed image filename. `app_settings` persists one global automatic-fetch preference and every fresh or upgraded library defaults it to off. Pending, ready, failed, retry, removed, next-attempt, and cache state survive process relaunch. Startup scans and reconciles every active note before opening a fail-closed drain gate for at most four deduplicated requests; it does not drain per page, and opt-in while the scan is running cannot bypass the gate. A failed startup reconciliation leaves fetching gated and surfaces a warning. Background work starts only after the initial timeline has hydrated.

Turning automatic fetching off is a barrier: the coordinator first flips its runtime gate and cancels fetch, DNS, redirect-validation, and URLSession work, then persists the off state; if persistence fails the runtime gate remains safely off. The database also rejects late cache commits while off. Canceled generations cannot mark a re-created association failed, and every status/cache mutation checks that eligible note associations still exist. Preview-only UI refreshes merge only `linkPreviews` into the latest in-memory note and discard superseded refresh generations, so they cannot restore stale note text or attachment state. Removing a card records suppression without modifying note text; editing the URL away removes its association and suppression, so a later re-add is eligible again. Permanent note deletion returns unreferenced preview-image filenames for post-commit removal under the same managed-file cleanup boundary as attachments.

Successful metadata is fresh for seven days. A stale success remains visible while one shared refresh is attempted; refresh failures keep the prior card and persist a request-key-wide 24-hour deadline so a newly associated note cannot bypass the shared backoff. Failed first fetches expose retry. Cache/image rows are retained while referenced and otherwise removed. Preview file reads, startup cleanup, and permanent-delete cleanup accept only non-symlink regular files whose names exactly match the generated lowercase UUID `.png` contract; startup deletes unreferenced managed files and clears missing-image pointers without discarding text metadata. Preview URL/title/description/site metadata is included in the FTS source, while **Has Link** is based on an active detected association and continues to match after a card is manually removed.

**Why:** Capture must remain durable and instant regardless of network state. Persisting association, work, preference, backoff, and shared cache state makes interruption and relaunch deterministic without coupling a note's existence to a remote site.

## ADR-009: Constrained URLSession transport with conservative SSRF rejection

**Decision:** Preview transport uses an injected raw `URLSession`/DNS boundary, never a web view. Only normalized HTTP and HTTPS URLs up to 2,048 characters are accepted; embedded credentials, fragments, invalid ports, single-label/special-use names, and unsafe literal addresses are rejected. Before the first connection and after each of at most five redirects, all initially observed A/AAAA answers must pass a conservative globally-routable predicate. Loopback, unspecified, private, shared, link-local, multicast, benchmarking, documentation, transition, mapped-private, and other special-use ranges/names are denied. DNS resolution, redirect validation, continuation publication, and task cancellation are race-safe. URLSession accumulation retains a one-shot terminal result until its checked continuation is published. Once DNS-SD has accepted its serial callback queue, reference deallocation and retained callback-context release are serialized onto that queue after any executing callback; setup failures before queue association use a separate direct cleanup path.

Sessions are ephemeral and disable cookie storage, credential storage, HTTP cookies, caches, proxy/PAC discovery, and connectivity waiting. Requests are clean unauthenticated GETs, non-server-trust authentication challenges are canceled, and no JavaScript is parsed or executed. Each resource has a 12-second total deadline and an 8-second per-request ceiling. HTML is limited to 1 MiB and `text/html` or `application/xhtml+xml`; up to two static image candidates are attempted, each limited to 2 MiB and a small accepted raster-type allowlist. Streamed bytes are bounded even without `Content-Length`. ImageIO accepts only known source types with at most 20 million declared pixels, renders only the first frame to a maximum 512-pixel static PNG, and rechecks the encoded size. Open Graph metadata is sanitized and length-bounded, with inert HTML title/meta/host/favicon fallbacks.

Supporting user-authored `http://` links requires ATS arbitrary-load permission; the only new App Sandbox entitlement is outbound network client access. HTTP and downgrade redirects are therefore plaintext and may disclose the requested path. The setting explains that any fetch reveals the user's IP address and destination to the site.

**Platform limit:** DNS validation is necessarily a preflight separate from URLSession's actual connection resolution. Public macOS APIs do not let the app pin URLSession to the validated address or inspect every connected peer, so DNS rebinding, late resolver answers, VPNs, transparent proxies, Network Extensions, and CFNetwork's internal decompression/buffering cannot be eliminated completely. The implementation validates every address and redirect observable through supported APIs, uses a short DNS quiet period to collect the initial answer set, disables explicit proxy/PAC settings, and enforces post-delivery byte limits. These limits must remain documented rather than being hidden by weakening test policy.

**Why:** Link metadata is hostile remote input and URLs may target local services. A small injectable transport permits deterministic tests while applying the strictest practical policy available above URLSession without browser state or script execution.

## ADR-010: Verified snapshots and pre-open journaled restore

**Decision:** `ArchiveService` creates each backup database with GRDB's SQLite online-backup API, never by copying a running database, WAL, or shared-memory file. It inspects the closed snapshot read-only, derives the complete managed-file requirement set from that snapshot, copies only strict managed paths through no-follow regular-file descriptors, and publishes a unique package only after a versioned sorted manifest, all SHA-256/size records, exact package shape, migration history, `PRAGMA integrity_check`, `PRAGMA foreign_key_check`, and database-owned file hashes agree. Files are synchronized as written; every modified nested package directory is then synchronized deepest-first before the package root and exclusive destination rename are synchronized. The published package is checked again without cancellation before success. Portable export uses the same snapshot boundary and directory-durability sequence so note rows and nested relationships are from one database point in time.

Manual backup/export destinations come from native panels and are used only while their sandbox extension is active. Automatic backup persists an app-scoped security-scoped folder bookmark. Destination roots must be regular non-symlink directories outside both the active library and its recovery control area. Partial and final names contain generated UUIDs, publication uses an exclusive rename, and no generated operation overwrites an existing path. Manifest paths are a closed allowlist with strict components and canonical-collision rejection; package enumeration rejects missing, additional, symbolic-link, and non-regular entries. Source file identity, size, and modification time are checked around streaming so a changed selection fails instead of producing a mixed success.

Restore is a pre-open crash state machine owned by `RestoreRecoveryCoordinator`. The recovery control directory is a deterministic sibling of the active library, not a descendant of either directory used in the switch, and therefore remains addressable across the swap. A selected package is first copied into a private quarantine and fully revalidated there. Its validated `library` then becomes a same-volume staged directory. Distinct old/new identity records are fsynced into each root before a version-1 journal is atomically published. If journal durability is uncertain, identity and staging cleanup occurs only when journal absence is established; otherwise both copies are preserved and UI pauses mutation until relaunch.

The durable states are `armed`, `trialStarted`, `rollbackRequested`, `cancellationRequested`, and `committed`. Before constructing `AppDatabase`, an armed launch uses Darwin `renameatx_np(..., RENAME_SWAP)` on directory descriptors and synchronizes their parent directory, then retains the old root as `rollback-library`. A trial is not committed merely because SQLite opens: a detached full integrity/foreign-key scan, timeline and draft hydration, attachment maintenance, and link-preview startup maintenance must complete. Failure requests rollback; process exit before confirmation is itself treated as an unsuccessful trial at the next launch. Rollback uses the same atomic swap before SQLite opens. Pre-open handling of unarmed staging, `cancellationRequested`, and `committed` validates identities and the selected winner without recursively deleting a large library. Those idempotent deletions run in the detached startup-confirmation phase while the single UI mutation gate and link-preview actor pause remain closed. Cleanup failure retains its typed canceled/committed outcome and requests relaunch; once `committed` is durably visible, the active winner is never described as rollbackable even when cleanup still needs another launch. Unknown identities, journals, files, or directory combinations fail closed and preserve both roots.

Automatic rotation runs only after the newly published package validates. The new verified automatic package is pinned, then the six newest other verified automatic packages are retained. Remaining verified automatic packages are eligible for removal; manual, incompatible, corrupt, partial, and unrelated items are not. A failure stops rotation and reports whether any older candidate was already removed, while the pinned seven-or-fewer set remains untouched.

Portable format version 1 consists of `manifest.json`, `export.json`, regenerated `notes.md`, and copied immutable originals under unique safe `attachments/` names. JSON keeps ascending `sortKey` chronology, all note timestamps and Active/Trash state, attachment identity/order/hash metadata nested under its note, and persisted URL/link-preview/cache timing metadata nested under its note. Markdown is deterministically generated from that decoded JSON and is checked byte-for-byte during validation. Original names remain metadata and labels; unsafe copied-name characters are replaced, canonical collisions append the stable logical attachment UUID, and a deterministic numeric suffix is the final fallback. Cached preview PNGs and attachment drafts are restorable backup state but intentionally are not portable-note content.

**Recovery invariants:** No active root is renamed before a quarantined copy has passed complete validation. The atomic switch always retains the prior root until restored startup commits. Any unconfirmed trial returns to the prior root before SQLite reopens. Any durability ambiguity preserves both copies and stops mutation instead of reporting success. Backup/export cancellation is observed before publication; final post-publication verification and restore arming are monotonic non-cancelable boundaries. Rotation never removes the newly created verified package or all known-good packages.

**Why:** SQLite can produce a transactionally consistent database snapshot, but it cannot atomically transact with managed files or an application relaunch. Closed inventories plus fail-closed verification make package consistency explicit. A sibling durable journal and same-volume atomic directory exchange reduce every supported interruption to an identifiable winner and recoverable counterpart without pretending a sequence of ordinary moves is atomic.

## ADR-011: Read-only health reporting and closed redacted support data

**Decision:** Phase 6 library health is an explicit user action implemented beside, not inside, mutation and startup-maintenance paths. It reads through the already-open `AppDatabase` queue and the same archive-inspector query body used for backup validation, requires the database path to retain its opened device/inode identity, requires exact shipped migration history, runs `PRAGMA integrity_check` and `PRAGMA foreign_key_check`, and derives managed-file requirements only from database-owned UUID filenames, byte counts, and stored hashes. The timeline mutation gate flushes the recoverable draft, blocks new note/archive/attachment/restore/support mutation, and pauses/drains the preview actor before inspection; it resumes preview work after the support operation finishes. Managed directories are opened component-by-component from the library root with `openat(O_DIRECTORY | O_NOFOLLOW)`. Referenced files and directory enumeration remain anchored to those descriptors, so substituted leaf or ancestor symlinks are never followed. Every referenced file must be ordinary; database-owned size/hash is checked when available, and pre/post file identity, size, and modification time must remain stable while hashing. Results contain only status and aggregate counts. No health-check code repairs, removes, renames, stages, or writes library data. Database scanning and file hashing run detached from the main actor; ambiguity is reported, never “repaired.”

Diagnostics have a closed schema: event and outcome enums, timestamps, and a fixed set of integer metrics. There is no arbitrary-message, URL, filename, path, error-description, or metadata-string API. Unified logging emits only event and outcome; an in-memory lock-protected ring retains at most 200 typed records. Backup, restore, portable-export, startup, health, and support-export outcomes use this boundary without copying existing user-facing error text.

Support-information format `com.selfdmnotes.support-information` version `1` is generated from app/build/OS/architecture constants, schema/migration constants, automatic-backup configured/last-success state, aggregate health, and the typed diagnostic ring. The payload cannot represent note/draft bodies, link URLs or preview text, original/managed filenames, attachment bytes/hashes, absolute paths, selected folder names, or bookmarks. Export writes a generated owner-only hidden partial with `O_EXCL | O_NOFOLLOW`, synchronizes it, and publishes to a new user-selected name with `RENAME_EXCL`; it refuses existing destinations and any destination inside the active library. Pre-publication low-disk, read-only, and denied-access failures remove the partial and leave existing data unchanged. Backups and portable exports remain intentionally private-content-bearing formats and are kept distinct in naming, UI, and documentation.

The Phase 6 UI uses native controls and system fonts, preserves Slack-compatible composer handling, exposes a visible shortcut reference, and adds first-run local-storage/backup/preview-privacy/no-sync guidance plus About/support surfaces. Sheets restore focus to their invoking header control, edited note, or composer as appropriate. Search completion, attachment state transitions, status/error banners, and health completion request VoiceOver announcements; rapidly changing archive progress exposes an accessibility value without repeatedly interrupting speech. Custom timeline scrolling continues to respect Reduce Motion. Preview-image disk read/decode now follows the existing detached `CGImage` thumbnail pattern rather than synchronously constructing `NSImage` from SwiftUI `body`.

**Why:** A consistency checker is trustworthy only if asking for information cannot turn a recoverable inconsistency into data loss. A closed support schema prevents later call sites from accidentally treating private user-facing errors as diagnostics. Exclusive durable publication and aggregate-only results make the support artifact useful without making it another archive export. Accessibility behavior remains localized to the established SwiftUI/AppKit focus and rendering boundaries instead of redesigning stable persistence or navigation architecture.
